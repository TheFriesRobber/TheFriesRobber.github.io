Spark / Yarn

关于Apache Spark在YARN上的资源请求，数据结构的一些notes

2020/04/15
SparkBigdata

问题

如何向一个拥有6个node，每个node有16个core和64GB内存大小的YARN Cluster申请资源，以最大化利用集群资源？并行度（partitions）设置多少合适？RDD中的数据用什么格式合适？

YARN集群资源申请

这里我们只考虑CPU和Memory（Disk和Network对Spark的性能影响也很大，不过Spark和YARN都没有提供主动管理他们的方式）。Apache Spark可以通过设置executor的数量，以及每个executor中的core和memory相关的config来申请集群资源。

Executor数量

指定一个Spark Application中executor的数量。支持动态的申请和释放executor，通过设置dynamic allocation相关参数，如spark.dynamicAllocation.enabled.

 // spark-submit
 --num-executors
 // spark-defaults.conf or SparkConf object
 spark.executor.instances

CPU

通过指定executor cores可以指定一个executor中task执行的并行度。

// spark-submit
--executor-cores
// spark-defaults.conf or SparkConf object
spark.executor.cores

--executor-cores = 5 意味着每个executor可以同时跑5个task。

Memory

同样通过指定executor memory可以指定一个executor中内存的大小限制。 内存太大会造成资源浪费，太小会导致内存中可处理的数据集小，要频繁的使用外存，影响performance。

// spark-submit
--executor-memory
// spark-defaults.conf or SparkConf object
spark.executor.memory

YARN中资源和Spark申请的资源对应

显然，Spark不能申请超过YARN中available的资源，有两个相关参数：

  • yarn.nodemanager.resource.memory-mb controls the maximum sum of memory used by the containers on each node.
  • yarn.nodemanager.resource.cpu-vcores controls the maximum sum of cores used by the containers on each node.

就executor core而言，一个core对应YARN中的一个vcore。memory相对有几点要考虑：

  • --executor-memory/spark.executor.memory控制的是executor heap的大小，但是JVM也会用到堆外内存（DirectByteBuffer）。因此，spark.yarn.executor.memoryOverhead这个参数也会被考虑进来，作为向YARN申请的总内存的一部分。默认max(384, .07 * spark.executor.memory)
  • YARN会对申请的内存取整。yarn.scheduler.minimum-allocation-mb和yarn.scheduler.increment-allocation-mb properties控制最小和increment的内存大小

另外还有几点需要考虑：

  • 我们需要考虑Application Master所需要的resource。在yarn-client下，默认需要一个vcore和1G内存。在yarn-cluster下，AM需要额外运行driver，所以通常会适当调大一些资源（--driver-memory & --driver-core）
  • 如果给executor分配了过大的内存，GC会比较严重。所以，64G是一个比较合适的上限对于单个executor
  • 5个core per executor是一个比较合适的上限（HDFS IO）
  • 避免一个executor只有一个core，因为就不能得到JVM多线程的好处。例如，广播变量需要被copy到所有executor中，多个thread/core可以共用内存中的广播变量，省去不必要的copy

再来看看问题

如何向一个拥有6个node，每个node有16个core和64GB内存大小的YARN Cluster申请资源，以最大化利用集群资源？

  • 首先，node manager中这两个yarn.nodemanager.resource.memory-mb 和 yarn.nodemanager.resource.cpu-vcores可以分别设为53 * 1024 = 64512MB和15。这里需要预留一些资源给OS和hadoop的进程，这里预留了一个vcore和1GB。
  • 这里我们其实会有一个非常直观的一个答案：--num-executors 6 --executor-cores 15 --executor-memory 63G。但是这个答案是有缺陷的。
    • —executor-memory只是executor heap的大小，如果加上executor memory overhead则会超过63G的可分配上限。
    • Application Master也需要一些资源，因此executor不能占据node的所有vcore。
    • 一个executor 15个core会导致HDFS IO的throughput下降
  • 一个更好的方案是--num-executors 17 --executor-cores 5 --executor-memory 19G
    • core：五个node中有三个executor，一个node有两个executor，并且AM运行在这个node上（剩余5个vcore可以分配）
    • memory：node中可分配63G，所以一个executor最多21G，算上overhead的0.07，21 * 0.07 = 1.47. 21 – 1.47 ~ 19

数据需要瘦身

Spark中数据流主要是有一条条记录组成，每条记录都有两种存在形式：未序列化的java object（内存中）和序列化的二进制（Disk中，网络传输）。 spark.serializer可以指定serializer（推荐使用kryo，org.apache.spark.serializer.KryoSerializer）

另外，我们还需要考虑使用一些数据格式（Avro, Parquet, Thrift, or Protobuf），第一可以减少data的size，第二可以利用格式的特性来加速数据处理，如columnar storage。这里就不做展开了。

回到问题

所以在提交spark application时，如何请求合适的resource（core，memory）对提高performance十分重要。同时，考虑到data会通过网络，disk在各个stage中流动，选择合适的serializer和数据格式也需要我们注意。
